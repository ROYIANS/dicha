### 2. File-based routing + `@tanstack/router-plugin/vite`

The current export from `@tanstack/router-plugin/vite` is **`tanstackRouter`** (camelCase). The old `TanStackRouterVite` name is deprecated — do not use it.

Confirmed plugin option **defaults** (from `router-generator/config.js`):
- `target` → default `"react"` (set explicitly anyway, per docs)
- `routesDirectory` → `"./src/routes"`
- `generatedRouteTree` → `"./src/routeTree.gen.ts"`
- `quoteStyle` → `"single"`, `semicolons` → `false`, `routeFileIgnorePrefix` → `"-"`
- `autoCodeSplitting` → unset (off); set `true` for per-route code splitting
- `enableRouteTreeFormatting` → `true`

```ts
// apps/web/vite.config.ts
import { defineConfig } from 'vite';
import { tanstackRouter } from '@tanstack/router-plugin/vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [
    // ⚠️ MUST come BEFORE @vitejs/plugin-react
    tanstackRouter({ target: 'react', autoCodeSplitting: true }),
    react(),
  ],
  server: {
    proxy: { '/api': { target: 'http://localhost:3000', changeOrigin: true } }, // see §6
  },
});
```

Generated files & layout (defaults):
```
apps/web/src/
├── routes/
│   ├── __root.tsx          # createRootRouteWithContext()(...) — shell + <Outlet/>
│   └── index.tsx           # createFileRoute('/')({ ... })
├── routeTree.gen.ts        # AUTO-GENERATED by the plugin — do not edit
└── main.tsx
```
- `routeTree.gen.ts` is written/updated by the plugin on `vite dev`/`vite build` (and via `@tanstack/router-cli` `tsr generate` if you want it without Vite). It is normally **committed** (so type-check works on a clean checkout) — add an ESLint/Prettier ignore for it. Optionally `.gitignore` it instead, but then CI must run codegen before `tsc`.
- `__root.tsx` uses `createRootRouteWithContext<RouterContext>()` (see §3); leaf routes use `createFileRoute('<path>')(...)`.

### 3. Typed router context carrying `queryClient`

`createRootRouteWithContext` is the current API (verified exported from `@tanstack/react-router`). Define the context type once; the plugin threads it through every route's `loader`/`beforeLoad`.

```ts
// apps/web/src/routes/__root.tsx
import { createRootRouteWithContext, Outlet } from '@tanstack/react-router';
import type { QueryClient } from '@tanstack/react-query';

export interface RouterContext {
  queryClient: QueryClient;
}

export const Route = createRootRouteWithContext<RouterContext>()({
  component: () => <Outlet />,
  // pendingComponent / errorComponent can also live here (see §5)
});
```

```ts
// apps/web/src/router.tsx
import { createRouter } from '@tanstack/react-router';
import { QueryClient } from '@tanstack/react-query';
import { routeTree } from './routeTree.gen';

export const queryClient = new QueryClient();

export const router = createRouter({
  routeTree,
  context: { queryClient },     // ← the literal pattern the task asks for
  defaultPreload: 'intent',     // §5
});

// Make the router fully type-safe app-wide via the Register interface
declare module '@tanstack/react-router' {
  interface Register {
    router: typeof router;
  }
}
```

```tsx
// apps/web/src/main.tsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import { RouterProvider } from '@tanstack/react-router';
import { QueryClientProvider } from '@tanstack/react-query';
import { router, queryClient } from './router';

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <QueryClientProvider client={queryClient}>
      <RouterProvider router={router} />
    </QueryClientProvider>
  </StrictMode>,
);
```

> Two valid context patterns:
> - **A (above, simplest, matches task):** create `queryClient` at module scope, pass into `createRouter({ context: { queryClient } })`, and feed the **same instance** to `QueryClientProvider`. `RouterProvider` does **not** re-pass context.
> - **B (placeholder):** `createRouter({ context: { queryClient: undefined! } })`, then `<RouterProvider router={router} context={{ queryClient }} />`. Use only when the client must be created inside a component. Pattern A is preferred here.

### 4. Loader-first data (no waterfall) — the canonical pattern

A shared `queryOptions` factory is the linchpin: the route `loader` warms the cache via `queryClient.ensureQueryData(...)`, and the component reads the **same** options via `useQuery(...)`, hitting the now-populated cache (no second fetch, no waterfall).

```ts
// apps/web/src/features/health/api.ts  (later via @ts-rest/react-query — see ts-rest research)
import { queryOptions } from '@tanstack/react-query';

export const healthQueryOptions = () =>
  queryOptions({
    queryKey: ['health'],
    queryFn: async () => {
      const res = await fetch('/api/health');     // §6 proxy
      if (!res.ok) throw new Error('health failed');
      return res.json() as Promise<{ status: string; db: 'up' | 'down' }>;
    },
  });
```

```tsx
// apps/web/src/routes/index.tsx
import { createFileRoute } from '@tanstack/react-router';
import { useQuery } from '@tanstack/react-query';
import { healthQueryOptions } from '../features/health/api';

export const Route = createFileRoute('/')({
  // loader runs before render; queryClient comes from typed router context (§3)
  loader: ({ context: { queryClient } }) =>
    queryClient.ensureQueryData(healthQueryOptions()),
  component: HomePage,
});

function HomePage() {
  // same options → reads loader-warmed cache, no refetch/waterfall
  const { data } = useQuery(healthQueryOptions());
  return <div>DB: {data?.db}</div>;
}
```

Notes:
- `ensureQueryData` is a **method on `QueryClient`** (not a top-level import); `queryOptions` and `useQuery` are exported from `@tanstack/react-query` (verified).
- If you'd rather suspend in the component, swap `useQuery`→`useSuspenseQuery(healthQueryOptions())` (also exported) — but with loader prefetch the plain `useQuery` already has data synchronously.
- This `xxxQueryOptions` factory **is** the loader-first contract called out in `architecture.md §2` / `state-management.md` (loader + component share one query def). `apps/web/src/features/*/api.ts` must export the factory, not just a hook.

### 5. Preload + route-level pending/error + `beforeLoad` (future-auth seam)

- **`defaultPreload: 'intent'`** on `createRouter` (§3) → hover/touchstart prefetches the route + its loader. Tune with `defaultPreloadStaleTime` (router won't re-run loaders for data still fresh in Query — set to `0` to delegate freshness entirely to Query's own `staleTime`).
- **Per-route boundaries:** `pendingComponent` (shown while loader pending) and `errorComponent` (catches loader/render errors). Router-wide fallbacks: `defaultPendingComponent` / `defaultErrorComponent` on `createRouter`. `architecture.md §2/§6` wants route-level pending/error + one app-root error boundary — put a `defaultErrorComponent` on the router and/or `errorComponent` on `__root.tsx`.

```ts
export const Route = createFileRoute('/')({
  loader: ({ context: { queryClient } }) => queryClient.ensureQueryData(healthQueryOptions()),
  pendingComponent: () => <div>Loading…</div>,
  errorComponent: ({ error }) => <div>Failed: {String(error)}</div>,
  component: HomePage,
});
```

- **`beforeLoad` auth seam (auth is OUT OF SCOPE now — just the seam):** `beforeLoad` runs **before** `loader` and receives `{ context, location }`. A future guard throws `redirect(...)`:

```ts
import { redirect } from '@tanstack/react-router';   // verified export
// FUTURE (architecture.md §3 BFF): not implemented in this scaffold
beforeLoad: ({ context, location }) => {
  if (!context.auth?.isAuthed) {
    throw redirect({ to: '/login', search: { redirect: location.href } });
  }
},
```
To wire it later: extend `RouterContext` (§3) with an `auth` field fed from the `/api/auth/me` Query (per `state-management.md`, auth-state is server state, not Zustand). No `beforeLoad` needed in M1 scaffold — just leave the context shape extensible.

### 6. Vite dev proxy `/api` → `http://localhost:3000` (same-origin for BFF cookies)

```ts
server: {
  proxy: {
    '/api': { target: 'http://localhost:3000', changeOrigin: true },
  },
},
```
This makes web (Vite dev server, default `:5173`) and NestJS same-origin in the browser, so the future `httpOnly; SameSite` session cookie works and CORS is avoided (`architecture.md §3/§6`).

**Path-prefix decision the implementer must make** — current `apps/api` (`apps/api/src/main.ts`) has **no global prefix**; its route is `GET /health` (not `/api/health`). The PRD wants the loader to fetch `/api/health`. So pick one:
- **(a)** add `app.setGlobalPrefix('api')` in `apps/api/src/main.ts` ⇒ backend serves `/api/health`; proxy forwards `/api/*` **untouched** (config above works as-is). Cleaner; keeps URLs identical dev/prod.
- **(b)** keep backend at root and **strip** the prefix in the proxy: add `rewrite: (p) => p.replace(/^\/api/, '')` so `/api/health` → `:3000/health`.

Option (a) is the more common monorepo choice (one URL shape everywhere); either satisfies the PRD. (Stated as the existing state + the seam, not a mandate.)

### 7. React 19 compatibility notes

- All four libs declare React 19 support: router peer `>=18 || >=19`, query peer `^18 || ^19`, devtools peers `>=19` — `react@19.2.7` satisfies all. ✓
- `@vitejs/plugin-react@6` requires **`vite ^8`** (hard peer). The **React Compiler** is opt-in via the optional `babel-plugin-react-compiler` peer — not required for this scaffold; skip it for M1.
- Entry stays `createRoot` from `react-dom/client` + `<StrictMode>`. StrictMode's double-invoke is fine with loader-first (loaders are de-duped through the Query cache).
- `@vitejs/plugin-react` (Babel) and the swc variant both work; the router plugin already runs a Babel transform on route files, so the Babel `plugin-react` pairs naturally. **Plugin order matters:** `tanstackRouter()` MUST be listed **before** `react()`.
- No deprecated React 19-removed APIs are used by this wiring (no legacy context, no string refs, no `ReactDOM.render`). Use React 19 `@types/react`/`@types/react-dom` to match.

### Mapping to this repo (Node 20, NodeNext base tsconfig)

- **tsconfig override required.** `tsconfig.base.json` is `module: NodeNext` / `moduleResolution: NodeNext`, which is wrong for a Vite browser app. `apps/web/tsconfig.json` should `extends` the base but **override**:
  ```jsonc
  {
    "extends": "../../tsconfig.base.json",
    "compilerOptions": {
      "module": "ESNext",
      "moduleResolution": "Bundler",   // not NodeNext, for Vite/browser
      "jsx": "react-jsx",
      "lib": ["ES2022", "DOM", "DOM.Iterable"], // base only has ES2022
      "types": ["vite/client"],
      "noEmit": true                    // Vite bundles; tsc is type-check only
    },
    "include": ["src"]
  }
  ```
  Keep `strict` + `noUnusedLocals/Parameters` from base — the generated `routeTree.gen.ts` is fine under them, but make ESLint/Prettier **ignore** `src/routeTree.gen.ts`.
- **`typecheck` task:** mirror `apps/api`'s pattern (`tsc -p tsconfig … --noEmit`). The generated route tree must exist before `tsc` — it's committed (recommended) or CI runs `tsr generate` first.
- **Node 20 / pnpm 9.12 / Vite 8:** Vite 8 supports Node 20. ✓ No engine conflict.
